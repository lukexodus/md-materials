## Testing Validation Errors in Fastify

Fastify's schema-based validation (powered by Ajv) rejects invalid requests before they reach your route handler. Testing these error paths is as important as testing happy paths — it confirms your schemas are correctly defined and that clients receive meaningful, well-structured error responses.

---

### Why Validation Error Testing Matters

When Fastify receives a request that fails schema validation, it short-circuits the handler and returns a `400 Bad Request` with a structured error body. Your tests need to confirm:

- The correct HTTP status code is returned
- The error body contains the expected shape
- The right fields are flagged
- Valid requests are still accepted (regression guard)

---

### Default Validation Error Response Shape

Fastify's default error response for validation failures follows this structure:

json

```json
{
  "statusCode": 400,
  "error": "Bad Request",
  "message": "body/email must match format \"email\""
}
```

The `message` field is generated by Ajv and uses JSON Pointer notation to identify the offending property.

**Key Points**

- `statusCode` — always `400` for validation failures (unless customized)
- `error` — the HTTP error label
- `message` — Ajv-generated description of the first validation failure
- Multiple errors are not surfaced by default; Ajv stops at the first failure unless `allErrors: true` is set

### Setting Up a Testable Route with Validation

js

```js
// app.js
import Fastify from 'fastify'

export function buildApp() {
  const app = Fastify()

  app.post('/users', {
    schema: {
      body: {
        type: 'object',
        required: ['username', 'email', 'age'],
        properties: {
          username: { type: 'string', minLength: 3 },
          email: { type: 'string', format: 'email' },
          age: { type: 'integer', minimum: 18 }
        },
        additionalProperties: false
      }
    }
  }, async (request, reply) => {
    return { created: true, user: request.body }
  })

  return app
}
```

**Key Points**

- `required` — enforces presence of fields
- `minLength` / `minimum` / `format` — enforce value constraints
- `additionalProperties: false` — rejects unexpected fields
- The handler is never reached when validation fails

### Testing with Node Test Runner (`node:test`)

js

```js
// test/validation.test.js
import { describe, it, before, after } from 'node:test'
import assert from 'node:assert/strict'
import { buildApp } from '../app.js'

describe('POST /users — validation errors', () => {
  let app

  before(async () => {
    app = buildApp()
    await app.ready()
  })

  after(async () => {
    await app.close()
  })

  it('returns 400 when body is missing entirely', async () => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      headers: { 'content-type': 'application/json' },
      body: JSON.stringify({})
    })

    assert.equal(res.statusCode, 400)
    const body = res.json()
    assert.equal(body.error, 'Bad Request')
  })

  it('returns 400 when email format is invalid', async () => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      payload: {
        username: 'alice',
        email: 'not-an-email',
        age: 25
      }
    })

    assert.equal(res.statusCode, 400)
    const body = res.json()
    assert.match(body.message, /email/)
  })

  it('returns 400 when age is below minimum', async () => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      payload: {
        username: 'bob',
        email: 'bob@example.com',
        age: 16
      }
    })

    assert.equal(res.statusCode, 400)
    const body = res.json()
    assert.match(body.message, /age/)
  })

  it('returns 400 when username is too short', async () => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      payload: {
        username: 'ab',
        email: 'ab@example.com',
        age: 20
      }
    })

    assert.equal(res.statusCode, 400)
    assert.match(res.json().message, /username/)
  })

  it('returns 400 when additional properties are present', async () => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      payload: {
        username: 'carol',
        email: 'carol@example.com',
        age: 22,
        role: 'admin'
      }
    })

    assert.equal(res.statusCode, 400)
  })

  it('returns 201 for a fully valid payload (regression guard)', async () => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      payload: {
        username: 'dave',
        email: 'dave@example.com',
        age: 30
      }
    })

    assert.equal(res.statusCode, 200)
    assert.equal(res.json().created, true)
  })
})
```

**Key Points**

- `app.inject()` sends requests in-process — no HTTP server required
- `payload` is auto-serialized to JSON by `inject()`; use `body` with manual `JSON.stringify` when you need to test malformed JSON
- `res.json()` parses the response body
- `assert.match(str, /pattern/)` is useful when the exact Ajv message wording may vary across versions [Inference]

### Testing with Tap (`@fastify/tap-compatible` / `node-tap`)

js

```js
import t from 'tap'
import { buildApp } from '../app.js'

t.test('validation errors', async (t) => {
  const app = buildApp()
  await app.ready()
  t.teardown(() => app.close())

  t.test('missing required field', async (t) => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      payload: { email: 'x@example.com', age: 21 } // missing username
    })

    t.equal(res.statusCode, 400)
    t.match(res.json(), { error: 'Bad Request' })
  })

  t.test('invalid type for age', async (t) => {
    const res = await app.inject({
      method: 'POST',
      url: '/users',
      payload: {
        username: 'elena',
        email: 'elena@example.com',
        age: 'twenty'
      }
    })

    t.equal(res.statusCode, 400)
    t.match(res.json().message, /age/)
  })
})
```

**Key Points**

- `t.match()` in Tap does a deep partial match — useful for checking subsets of the response body
- `t.teardown()` handles cleanup even when assertions fail

### Testing Malformed JSON Bodies

Sending syntactically invalid JSON triggers a parse error, not a schema validation error. The status code is still `400`, but the message differs.

js

```js
it('returns 400 for malformed JSON', async () => {
  const res = await app.inject({
    method: 'POST',
    url: '/users',
    headers: { 'content-type': 'application/json' },
    body: '{ username: alice }' // invalid JSON — unquoted key
  })

  assert.equal(res.statusCode, 400)
  // message will reference JSON parse failure, not schema
})
```

**Key Points**

- Do not use `payload` here — use raw `body` with the `content-type` header manually set
- `payload` causes `inject()` to serialize the value as valid JSON automatically
- The resulting error message is distinct from Ajv errors; avoid asserting on the exact string unless you control the error handler [Inference]

### Testing with `allErrors: true`

By default, Ajv stops at the first validation failure. If you configure Ajv with `allErrors: true`, the response includes all failures at once.

js

```js
const app = Fastify({
  ajv: {
    customOptions: { allErrors: true }
  }
})
```

With `allErrors: true`, the default Fastify error response still contains only one combined `message` string. To expose all individual errors, a custom error handler or a plugin like `fastify-ajv-compiler` with custom serialization is needed.

**Example** — testing that multiple fields are invalid (with `allErrors: true` and a custom error handler):

js

```js
// custom error handler that exposes all Ajv validation errors
app.setErrorHandler((error, request, reply) => {
  if (error.validation) {
    return reply.status(400).send({
      statusCode: 400,
      errors: error.validation // array of Ajv error objects
    })
  }
  reply.send(error)
})
```

js

```js
it('surfaces all validation errors simultaneously', async () => {
  const res = await app.inject({
    method: 'POST',
    url: '/users',
    payload: {
      username: 'x',       // too short
      email: 'bad',        // bad format
      age: 10              // below minimum
    }
  })

  assert.equal(res.statusCode, 400)
  const { errors } = res.json()
  assert.ok(Array.isArray(errors))
  assert.ok(errors.length >= 3)
})
```

**Key Points**

- `error.validation` is the raw Ajv errors array attached by Fastify
- `error.validationContext` indicates where the failure occurred: `'body'`, `'params'`, `'query'`, or `'headers'`
- Behavior of `allErrors` depends on your Ajv version and configuration [Unverified — verify against your specific Ajv and Fastify versions]

### Testing Query and Params Validation

Validation applies to all schema targets, not just `body`.

js

```js
app.get('/items/:id', {
  schema: {
    params: {
      type: 'object',
      properties: {
        id: { type: 'integer' }
      },
      required: ['id']
    },
    querystring: {
      type: 'object',
      properties: {
        limit: { type: 'integer', minimum: 1, maximum: 100 }
      }
    }
  }
}, async (request) => {
  return { id: request.params.id, limit: request.query.limit }
})
```

js

```js
it('returns 400 when :id is not an integer', async () => {
  const res = await app.inject({
    method: 'GET',
    url: '/items/abc'
  })

  assert.equal(res.statusCode, 400)
  assert.match(res.json().message, /params/)
})

it('returns 400 when limit exceeds maximum', async () => {
  const res = await app.inject({
    method: 'GET',
    url: '/items/5?limit=500'
  })

  assert.equal(res.statusCode, 400)
})
```

**Key Points**

- Path parameters arrive as strings; Fastify coerces them to the declared type when `coerceTypes` is enabled (which it is by default in Ajv) [Inference — verify with your config]
- `error.validationContext` will be `'params'` or `'querystring'` respectively

### Using `error.validation` in Custom Error Handlers

js

```js
app.setErrorHandler((error, request, reply) => {
  if (error.validation) {
    const context = error.validationContext // 'body' | 'params' | 'querystring' | 'headers'
    const failures = error.validation.map(e => ({
      field: e.instancePath || e.params?.missingProperty,
      message: e.message
    }))

    return reply.status(400).send({
      statusCode: 400,
      validationContext: context,
      failures
    })
  }

  reply.send(error)
})
```

js

```js
it('returns structured failures from custom error handler', async () => {
  const res = await app.inject({
    method: 'POST',
    url: '/users',
    payload: { username: 'ab', email: 'bad', age: 10 }
  })

  assert.equal(res.statusCode, 400)
  const body = res.json()
  assert.equal(body.validationContext, 'body')
  assert.ok(Array.isArray(body.failures))
})
```

### Common Pitfalls

**Using `body` instead of `payload` in `inject()`**
If you pass `payload`, `inject()` serializes it to JSON automatically. If you pass `body`, it must already be a string and the `content-type` header must be set manually. Mixing these causes unexpected test behavior. [Inference — behavior may vary; verify against your Fastify version]

**Asserting on exact Ajv message strings**
Ajv message wording can change between minor versions. Prefer `assert.match(msg, /keyword/)` over strict equality on the full message string.

**Not calling `await app.ready()`**
If you skip `app.ready()`, plugins and schema compilation may not have completed. Subsequent `inject()` calls may behave unexpectedly. [Inference]

**Forgetting to close the app**
Always call `app.close()` in teardown. Unclosed Fastify instances can cause open handle warnings in test runners.
