## Upgrading HTTP to WebSocket

The WebSocket protocol begins as a standard HTTP/1.1 request. The client sends a specially crafted `GET` request with an `Upgrade` header, and the server responds with `101 Switching Protocols`. After that exchange, the TCP connection is repurposed as a persistent, full-duplex WebSocket channel.

Understanding this upgrade mechanism is important for debugging connection failures, implementing custom authentication, and configuring proxies and load balancers correctly.

---

### The HTTP Upgrade Handshake

#### Client request

http

```http
GET /ws HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Origin: https://example.com
```

#### Server response

http

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

**Key Points**

- `Upgrade: websocket` and `Connection: Upgrade` are mandatory on both sides
- `Sec-WebSocket-Key` is a random Base64-encoded 16-byte value generated by the client
- `Sec-WebSocket-Accept` is derived by the server: it concatenates the key with the fixed GUID `258EAFA5-E914-47DA-95CA-C5AB0DC85B11`, SHA-1 hashes the result, and Base64-encodes it
- The server never generates the key — it only validates and transforms it
- After `101`, the HTTP framing ends; WebSocket frames begin on the same TCP connection

### What Fastify Does Internally

When `@fastify/websocket` is registered, it hooks into Fastify's underlying `http.Server` `upgrade` event. Fastify does not handle the upgrade itself — it delegates to the `ws` library's `WebSocket.Server`, which manages the handshake validation and TCP socket hand-off.

**Key Points**

- The `upgrade` event fires on `http.Server`, not on any Fastify route handler
- `@fastify/websocket` intercepts this event before it would otherwise be ignored
- If `@fastify/websocket` is not registered, upgrade requests fall through and the TCP connection is destroyed by Node.js by default [Inference — verify against your Node.js version]

### Performing the Upgrade Manually

In some situations — custom protocols, raw TCP control, or gradual migration — you may need to handle the upgrade yourself without `@fastify/websocket`.

#### Accessing the raw `upgrade` event

js

```js
import Fastify from 'fastify'
import { WebSocketServer } from 'ws'
import { createServer } from 'node:http'

const app = Fastify()
await app.ready()

const wss = new WebSocketServer({ noServer: true })

// Access the underlying Node.js HTTP server
app.server.on('upgrade', (request, socket, head) => {
  // Only upgrade requests to /ws
  if (request.url !== '/ws') {
    socket.destroy()
    return
  }

  wss.handleUpgrade(request, socket, head, (ws) => {
    wss.emit('connection', ws, request)
  })
})

wss.on('connection', (socket, request) => {
  socket.send('Connected via manual upgrade')

  socket.on('message', (msg) => {
    socket.send(`Echo: ${msg.toString()}`)
  })
})

await app.listen({ port: 3000 })
```

**Key Points**

- `noServer: true` tells `ws` not to create its own HTTP server — it only handles upgrades when explicitly called via `handleUpgrade()`
- `app.server` is the underlying `http.Server` instance Fastify wraps
- `socket.destroy()` rejects the upgrade for unrecognized paths — the client receives a TCP RST; no HTTP response is sent
- `head` is a `Buffer` containing any bytes already received past the HTTP headers on the socket — pass it through to `handleUpgrade()` unchanged

### Rejecting an Upgrade with an HTTP Response

`socket.destroy()` gives the client no readable response. To send a proper HTTP error before closing:

js

```js
app.server.on('upgrade', (request, socket, head) => {
  if (!isAuthorized(request)) {
    socket.write(
      'HTTP/1.1 401 Unauthorized\r\n' +
      'Content-Type: text/plain\r\n' +
      'Connection: close\r\n' +
      '\r\n' +
      'Unauthorized'
    )
    socket.destroy()
    return
  }

  wss.handleUpgrade(request, socket, head, (ws) => {
    wss.emit('connection', ws, request)
  })
})
```

**Key Points**

- At this point the connection is a raw TCP socket, not an HTTP response object — you write raw HTTP text manually
- The response must end with `\r\n\r\n` (blank line) to terminate the headers
- Call `socket.destroy()` after writing — do not leave the socket open
- Browsers will surface this as a failed WebSocket connection; the status code is not directly accessible via the browser `WebSocket` API [Inference]

### Reading Upgrade Request Headers

The `request` object in the `upgrade` event is a standard `http.IncomingMessage`:

js

```js
app.server.on('upgrade', (request, socket, head) => {
  const url = new URL(request.url, 'http://localhost')
  const token = url.searchParams.get('token')
  const origin = request.headers['origin']
  const protocol = request.headers['sec-websocket-protocol']

  console.log({ token, origin, protocol })
})
```

**Key Points**

- Query parameters must be parsed manually from `request.url` — Fastify's query parser is not applied at the upgrade level
- `sec-websocket-protocol` contains the subprotocol the client proposes (e.g. `graphql-ws`, `chat`)
- `origin` should be validated to prevent cross-origin WebSocket abuse

### Subprotocol Negotiation

Clients can propose one or more subprotocols via the `Sec-WebSocket-Protocol` header. The server selects one and echoes it back.

#### Client (browser)

js

```js
const ws = new WebSocket('ws://localhost:3000/ws', ['v1', 'v2'])
```

#### Server — manual handling

js

```js
app.server.on('upgrade', (request, socket, head) => {
  const proposed = request.headers['sec-websocket-protocol']
    ?.split(',')
    .map(s => s.trim()) ?? []

  const supported = ['v1']
  const selected = proposed.find(p => supported.includes(p))

  if (!selected) {
    socket.write('HTTP/1.1 426 Upgrade Required\r\n\r\n')
    socket.destroy()
    return
  }

  wss.handleUpgrade(request, socket, head, (ws) => {
    // Inform ws of the selected protocol
    wss.emit('connection', ws, request)
  })
})
```

#### Server — via `@fastify/websocket` handler options

js

```js
app.get('/ws', {
  websocket: true,
  wsOptions: {
    handleProtocols: (protocols, request) => {
      if (protocols.has('v1')) return 'v1'
      return false // reject
    }
  }
}, (socket, request) => {
  socket.send('Protocol negotiated')
})
```

**Key Points**

- `handleProtocols` receives a `Set` of proposed protocol strings and the `IncomingMessage`
- Return the selected protocol string, or `false` to reject
- If rejected, `ws` closes the connection [Inference — verify callback signature against your `ws` version]
- The negotiated protocol is accessible on the connected socket as `socket.protocol`

### Origin Validation

WebSocket connections from browsers include an `Origin` header. Servers should validate it to prevent cross-site WebSocket hijacking (CSWSH).

js

```js
const ALLOWED_ORIGINS = new Set([
  'https://example.com',
  'https://app.example.com'
])

app.server.on('upgrade', (request, socket, head) => {
  const origin = request.headers['origin']

  if (!ALLOWED_ORIGINS.has(origin)) {
    socket.write('HTTP/1.1 403 Forbidden\r\n\r\n')
    socket.destroy()
    return
  }

  wss.handleUpgrade(request, socket, head, (ws) => {
    wss.emit('connection', ws, request)
  })
})
```

**Key Points**

- Non-browser clients (CLI tools, server-to-server) may not send `Origin` at all — decide whether to allow or reject absent origins based on your use case
- `Origin` can be spoofed by non-browser clients; treat it as a browser-level protection, not a security boundary for all clients [Inference]
- For stricter control, combine origin validation with token authentication

### Handling Upgrades in a Fastify Plugin

Encapsulating upgrade logic in a plugin keeps it organized and testable:

js

```js
// src/plugins/raw-websocket.js
import fp from 'fastify-plugin'
import { WebSocketServer } from 'ws'

async function rawWebSocketPlugin(app, opts) {
  const wss = new WebSocketServer({ noServer: true })

  app.server.on('upgrade', (request, socket, head) => {
    const url = new URL(request.url, 'http://localhost')

    if (url.pathname !== (opts.path ?? '/ws')) {
      socket.destroy()
      return
    }

    wss.handleUpgrade(request, socket, head, (ws) => {
      wss.emit('connection', ws, request)
    })
  })

  wss.on('connection', (socket, request) => {
    opts.handler?.(socket, request)
  })

  app.addHook('onClose', async () => {
    await new Promise((resolve, reject) => {
      wss.close((err) => (err ? reject(err) : resolve()))
    })
  })
}

export default fp(rawWebSocketPlugin)
```

js

```js
// usage
await app.register(rawWebSocketPlugin, {
  path: '/ws/raw',
  handler: (socket, request) => {
    socket.send('Raw connection established')
    socket.on('message', (msg) => socket.send(msg.toString()))
  }
})
```

**Key Points**

- `app.addHook('onClose', ...)` ensures the `ws.Server` is closed when Fastify shuts down
- `fp()` unwraps encapsulation so `app.server` access propagates correctly [Inference]
- `opts.path` makes the plugin reusable across different routes

### HTTPS / WSS Upgrades

WebSocket over TLS (`wss://`) follows the same upgrade handshake, but the TCP connection is wrapped in TLS first. No changes to the WebSocket handshake logic are needed — TLS termination happens at the transport layer.

js

```js
import { createServer } from 'node:https'
import { readFileSync } from 'node:fs'

const app = Fastify({
  https: {
    key: readFileSync('./server.key'),
    cert: readFileSync('./server.cert')
  }
})
```

**Key Points**

- `app.server` is an `https.Server` when TLS is configured — `upgrade` events work identically
- In production, TLS is typically terminated at a reverse proxy (nginx, AWS ALB) rather than in Node.js directly [Inference]
- Proxies must be configured to pass `Upgrade` and `Connection` headers through — stripping them breaks the WebSocket handshake

### Proxy and Load Balancer Configuration

WebSocket upgrades require specific proxy configuration that differs from regular HTTP.

#### nginx

nginx

```nginx
location /ws {
    proxy_pass http://localhost:3000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "Upgrade";
    proxy_set_header Host $host;
    proxy_read_timeout 3600s;
}
```

**Key Points**

- `proxy_http_version 1.1` is required — WebSocket is not supported over HTTP/1.0
- `proxy_set_header Upgrade` and `Connection` must be forwarded explicitly — nginx strips hop-by-hop headers by default
- `proxy_read_timeout` should be set to a high value (or `0` for no timeout) — WebSocket connections are long-lived and will be dropped if the proxy times them out

#### AWS ALB

- ALB supports WebSocket natively with no additional configuration [Unverified — verify against current AWS documentation]
- Ensure target group health checks use HTTP, not WebSocket paths

### Upgrade Failure Scenarios

| Scenario | Outcome |
| --- | --- |
| Missing `Upgrade: websocket` header | Server treats as plain HTTP `GET`; route handler responds normally |
| Missing `Sec-WebSocket-Key` | `ws` rejects the handshake with `400 Bad Request` |
| Unsupported `Sec-WebSocket-Version` | `ws` responds with `426 Upgrade Required` and lists supported versions |
| Path not handled by any WebSocket route | `@fastify/websocket` destroys the socket; client sees connection refused |
| TLS mismatch (`ws://` vs `wss://`) | TCP connection fails before HTTP headers are sent |
| Proxy strips `Upgrade` header | Server sees plain HTTP request; no upgrade occurs |

### Upgrade Handshake Verification

The `Sec-WebSocket-Accept` derivation, for reference:

js

```js
import { createHash } from 'node:crypto'

const GUID = '258EAFA5-E914-47DA-95CA-C5AB0DC85B11'

function deriveAcceptKey(clientKey) {
  return createHash('sha1')
    .update(clientKey + GUID)
    .digest('base64')
}

// Example
const key = 'dGhlIHNhbXBsZSBub25jZQ=='
console.log(deriveAcceptKey(key))
// s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

**Key Points**

- This derivation is performed by the `ws` library automatically — you do not implement it manually in normal usage
- It exists to confirm the server understood the upgrade request and is not simply echoing arbitrary headers
- It does not provide security — it is an identity check, not authentication
