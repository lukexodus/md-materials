# Executive Summary

We surveyed a range of PaaS and cloud platforms offering free or low-cost hosting for multi-container Docker Compose apps. Our top recommendations for a small 3-service project are **Koyeb**, **Render**, and **Fly.io**. Koyeb has explicit Docker Compose support via a special Docker image and offers a usable free instance (512 MB RAM, autosleeps)【6†L132-L140】【11†L395-L404】. Render provides an easy-to-use free “Hobby” tier (750 hr/month, 512 MB RAM, spins down on idle) with native Docker support and managed DNS/TLS【13†L278-L282】【13†L236-L243】. Fly.io gives a generous free allowance (up to three 256 MB “shared-cpu-1x” VMs and 100 GB/mo bandwidth) and supports running a Compose app as a single multi-container “Machine”【45†L349-L357】. All three integrate Git-based deployments and offer custom domains with automatic TLS. In contrast, general-purpose clouds (AWS, GCP, Azure) only indirectly support Compose via VMs or container services and have steeper learning curves or limited free quotas. Our detailed comparison (below) shows each provider’s Compose support, free tier limits, setup workflow, storage, networking, scaling, and example deploy commands. Below we summarize each provider’s key features and steps.

- **Koyeb:** Koyeb is designed for “serverless containers”. It can run your multi-container app with minimal changes: you add a `Dockerfile.koyeb` that uses `FROM koyeb/docker-compose`, and Koyeb will launch all Compose-defined containers【6†L119-L128】. You get 0.1 vCPU / 512 MB RAM free, scaling to zero after 1h idle【11†L395-L404】. Setup is via the Koyeb Dashboard (Git integration) or CLI (`koyeb create stack` etc.)【6†L143-L153】【35†L85-L93】. It offers one free region and no free persistent storage【11†L397-L404】. Custom domains and Let’s Encrypt TLS are supported. Pros: true Docker Compose support, easy CI/CD integration; Cons: very limited free resources, no free volumes, free instance can’t act as a worker【11†L397-L404】.

- **Render:** Render provides a hobby (free) tier for web services (512 MB, 750 hr/mo, sleeps after 15 min idle)【13†L246-L254】【13†L278-L282】. You cannot feed a `docker-compose.yml` directly; instead, you define each service in the Render console or `render.yaml`: e.g. one “Web Service” for your backend, another for frontend, plus a managed Postgres service. Services run with private networking and can connect via env vars【32†L197-L206】【32†L243-L252】. Dockerfiles or public images are supported【15†L191-L200】. Setup is as simple as linking your GitHub repo and clicking “New Web Service” in the dashboard. Free services get a `*.onrender.com` domain, and Render provides auto TLS for custom domains【13†L236-L243】. Drawbacks: free tier has limited RAM and no free persistent disk; uses multiple Render services instead of one Compose file. 

- **Fly.io:** Fly lets you run Docker containers globally on small VMs. Its free tier (for legacy accounts) includes up to three *shared-cpu-1x* VMs (1×256 MB each), 3 GB of volume storage, and 100 GB of monthly egress in NA/EU【45†L349-L357】. Fly supports multi-container “Machines” by adding a `[build.compose]` section in `fly.toml`【40†L125-L134】. E.g. a single `fly deploy` can build your app’s service and start sidecars (like a Redis container) in the same VM【40†L125-L134】. Setup is via the `flyctl` CLI (`fly launch`, then `fly deploy`). Networking is powerful: every Fly app gets an Anycast IPv6 (and optional static IPv4) with free Let’s Encrypt certificates. Services can communicate internally by “container name.internal”. Fly’s paid pricing starts at about \$2/month for a 256 MB VM【45†L430-L438】. Pros: true Compose support, global distribution, free TLS, persistent volumes; Cons: very small free RAM per service, legacy billing complexity.

- **Railway:** Railway uses a novel “services” model. You transform each Compose service into a Railway service (or managed DB). For example, a `redis:` service in Compose becomes a Railway Redis database. The official guide walks through mapping Compose elements to Railway concepts【22†L369-L378】. After wiring up services, Railway handles internal DNS (`my-service-name.railway.internal`) and public endpoints. The free plan is basically a \$5 credit (~30 days) with up to 1 vCPU/0.5 GB per service【24†L25-L30】. It’s quite limited, but very easy: just connect your GitHub repo and deploy with `railway up` or via web. Custom domains and TLS are supported. Good for trial/hobby apps, but beyond the trial you quickly hit resource limits.

- **Google Cloud Run (or Cloud Run Jobs):** Cloud Run can host containerized services serverlessly. It has a generous free tier (2 million requests/mo plus 180k CPU-seconds and 360k GB-seconds free, with 1 GB outbound data free)【54†L53-L54】【56†L83-L86】. You would deploy each Compose service as a separate Cloud Run service (e.g. one service per container) and let them talk over HTTP or Pub/Sub. Setup is via `gcloud run deploy`. Cloud Run scales to zero, supports custom domains with managed SSL, and has wide global coverage. However, there’s no direct Docker Compose support; you must manage each service separately. Persistent storage must be external (Cloud Storage, managed DB). Cloud Run is pay-as-you-go after free quotas: roughly \$0.000018 per vCPU-second and \$0.000002 per GiB-second【56†L170-L178】. Example deploy: `gcloud run deploy web-service --image gcr.io/my-project/web --platform managed --region us-central1` and similarly for other services. Good for stateless microservices; less so if you rely on Compose’s private Docker networking model.

- **Other Clouds (AWS/GCP/Azure Compute):** The major clouds offer free VMs (e.g. AWS EC2/t2.micro, GCP f1-micro, Oracle Ampere VMs, Azure B1S) that can run Docker Compose manually. For example, a free EC2 micro (1 GB RAM) could run your stack. AWS also has ECS/Fargate with a Compose CLI plugin (but no free Fargate). Google GKE (Autopilot) offers one free cluster, but you still pay for nodes. Oracle Cloud’s Always Free tier gives you two 1-CPU VMs and free Kubernetes (OKE). These options are powerful but require much more ops work (OS maintenance, CLI commands, networking config) and don’t offer the streamlined Docker-Compose workflows of PaaS. They were omitted from the table for brevity, but are always an option (e.g. “spin up free VM + `docker-compose up`”).

Each platform has trade-offs:

- **Koyeb:** Best if you want the closest thing to “lift-and-shift” Docker Compose (requires minimal changes). Excellent for quick prototypes with Compose. However, the free tier is very constrained (essentially a tiny “function”) and not suited for heavier workloads. It’s easy for Python/Node apps as shown in Koyeb’s tutorial【6†L100-L110】, and you get a simple CLI or UI deployment.

- **Render:** Great developer experience and free tier convenience. It’s very easy to spin up static sites, web services, background workers, and managed DBs. The limitation is splitting your Compose into separate services – more manual work. But you gain professional features (managed databases, metrics, team support). Free services sleeping after 15 min idle can be annoying for APIs. 

- **Fly.io:** Very attractive for global, multi-container deployment. If your app is small enough (fits under ~256–512 MB per service), you get a free dynamic edge network plus native Compose support【40†L125-L134】. The learning curve is moderate (you use `flyctl`), but examples like deploying a web + Redis as one VM are well-documented【40†L125-L134】. Beware that Fly’s legacy free quotas require a credit card and some forum accounts; newly created orgs go on “pay-as-you-go” with no free allowance (but still free certs/IPv6). 

- **Railway:** Very easy to map a Docker Compose app to Railway if you’re okay with its resource limits. It shines if your Compose just needs a web + a DB/Redis – Railway manages the DB for you. But after the $5 trial, you must pay per-use, and free limits are tiny. 

- **Google Cloud Run:** Not a PaaS by Compose but worth mentioning. Its generous free tier and auto-scaling makes it an option, but you must architect your Compose app as independent services talking over HTTP or pub/sub. It supports custom domains/TLS and global regions. For example, you might deploy your `web` container to Cloud Run and have a separate managed Cloud SQL or Firestore for state. It’s more involved to set up than the above PaaS, so we recommend it only if you’re already on GCP. 

Other cloud options (AWS ECS/EKS/EC2, Azure Container Instances/Apps) similarly require more setup and often have minimal or no free usage for containers, so they’re less ideal for a quick low-budget start. Likewise, container platforms like **Heroku** have discontinued free dynos. 

In summary, for a hobby or small project with three Docker containers, we recommend **Koyeb, Render, or Fly.io**:

- **Koyeb** for direct Compose support and a truly serverless feel (auto-sleep, easy CLI) – *ideal for quick prototyping of Compose apps*【6†L132-L140】. 
- **Render** for the friendliest UI/UX and generous free usage (750 hrs/mo, built-in DB, simple Git deploy) – *great for web apps with moderate traffic*【13†L278-L282】.
- **Fly.io** if you need a highly distributed setup or native Compose; it’s the cheapest for always-on tiny apps and provides global autoscaling. 

Each can run a 3-service stack; choice depends on which constraints (RAM, uptime, regions) matter most. All three support Git-based CI/CD, environment variables, persistent storage (on paid plans), custom domains, and managed HTTPS out of the box【13†L236-L243】【45†L427-L434】.
